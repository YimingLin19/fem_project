# review.md

本文件定义本项目中 AI 生成代码或 AI 辅助修改代码时的审查规则。

## 审查目标

本项目属于有限元教学/实验代码，包含数值计算、边界条件、网格读取、装配、求解和后处理等环节。代码审查的目标是尽早识别以下问题：

- 明显错误的数值逻辑
- 破坏已有架构和主流程的实现
- 与现有接口不兼容的改动
- 隐蔽的边界条件错误
- 可能影响结果可信度的回归风险

## 审查原则

- 审查优先关注正确性、架构一致性和风险，而不是风格性意见。
- 审查必须结合本项目的实际特点进行，不应照搬通用 Web 项目或普通 CRUD 项目的标准。
- 审查时必须默认 AI 生成代码存在潜在错误，不能因为代码“看起来完整”就降低警惕。
- 审查结论应区分“必须修复的问题”和“可以后续讨论的建议”。

## 必查项

### 1. 正确性与数值逻辑

- 单元刚度、载荷、边界条件和后处理公式是否与现有实现保持一致。
- 是否错误改变了自由度顺序、节点顺序、单元节点映射或矩阵维度。
- 是否引入了错误的坐标系假设、积分点假设或材料参数解释。
- 是否存在明显的符号错误、漏项、重复装配或量纲不一致问题。

### 2. 架构一致性

- 改动是否遵守 `docs/project_guides/architecture.md`。
- 是否绕过了既有主链路：`mesh_io -> mesh/dof_manager -> stiffness -> assemble -> boundary -> solve -> post`。
- 是否把已有模块职责打散、复制，或额外引入平行实现。
- 是否在不必要的情况下改动了核心接口、核心数据结构或模块边界。

### 3. 接口兼容性

- 新代码是否与现有示例脚本的调用方式兼容。
- 是否破坏了当前基于 `src/fem` 的导入方式。
- 是否错误假设项目已经是可安装包。
- 涉及运行说明时，是否正确保留 `PYTHONPATH=src` 或等价配置前提。

### 4. 边界条件与异常路径

- 是否正确处理了位移约束、集中力、体力、边力/面力和重力。
- 是否考虑了空输入、缺失材料参数、非法单元拓扑、零面积、零体积、奇异矩阵等情况。
- 是否新增了静默失败路径，或把本应报错的问题吞掉。

### 5. 回归风险

- 改动是否可能影响现有 `examples` 中的求解流程。
- 是否可能改变既有结果文件格式、字段名或导出路径约定。
- 是否可能在 2D/3D、低阶/高阶单元之间引入不一致行为。

## 关于测试与验证

由于本项目的特殊性，代码验证通常需要单独完成，可能依赖额外数值对比、算例复现、结果曲线或人工核对。因此：

- 在 review 阶段，不要求必须同时补测试。
- 在 review 阶段，也不要求必须已经完成数值验证。
- 不能仅因为“没有测试”或“暂未验证”就否定一次代码审查的有效性。

但审查者仍然需要明确指出：

- 这次审查是否发现必须修复的问题。
- 哪些部分仍需要后续单独验证。
- 哪些结果风险无法仅靠静态代码审查消除。

## 审查输出要求

审查结论建议分为两类：

- Findings
  明确的问题、风险、逻辑错误、架构冲突、兼容性问题
- Follow-up validation
  需要后续单独验证的内容，但不作为此次 review 必须阻塞项

如果没有发现明确问题，也应明确写出：

- 本次 review 未发现显著代码问题
- 但哪些数值行为仍建议单独验证

## 审查时的禁止事项

- 不要把“建议补测试”混同为“代码必然错误”。
- 不要在未核对架构和主流程时，仅凭局部代码片段下结论。
- 不要把与项目现有模式不一致的个人偏好当成必须修改的问题。
- 不要在 `review.md` 或 `architecture.md` 不完整时，声称已经完成严格的完整审查。
