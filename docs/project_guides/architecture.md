# architecture.md

本文件定义本项目的架构约束，用于指导后续功能扩展、缺陷修复和代码审查。

## 项目定位

本项目是一个面向教学和实验的有限元代码库。

它的组织方式不是“完整应用 + 统一命令行入口”，而是：

- `src/fem` 提供核心库能力
- `examples` 提供具体算例脚本和使用入口

因此，架构上的核心要求不是抽象出复杂框架，而是保持主求解链路清晰、模块职责稳定、示例调用方式直观。

## 总体结构

当前仓库的核心结构如下：

- `src/fem/`
  有限元核心库
- `examples/`
  示例脚本、网格文件、材料文件
- `docs/project_guides/`
  协作规范、审查规则、架构约束

## 运行与导入约束

- 当前项目不是可直接安装的 Python 包。
- 项目没有 `pyproject.toml` 或 `setup.py`。
- 默认情况下直接执行 `python -c "import fem"` 会失败。
- 示例和开发流程依赖 `PYTHONPATH=src`，或在 IDE 中将 `src` 标记为源码目录。

这意味着：

- 不要在没有充分理由的情况下假设项目已包化安装。
- 不要在文档、示例或协作说明中省略这一前提。
- 如果未来要做包化改造，应视为单独的架构变更，而不是零散顺手修改。

## 核心主流程

本项目的主流程是固定的，新增功能和修复应优先挂接到这条链路上：

`mesh_io -> mesh/dof_manager -> stiffness -> assemble -> boundary -> solve -> post`

对应职责如下：

### 1. `mesh_io`

负责网格和材料输入。

- 读取 Abaqus `.inp`
- 读取部分 CSV 格式网格
- 将材料参数写入单元属性
- 根据输入构造对应的网格对象

### 2. `mesh` / `dof_manager`

负责核心数据结构和自由度编号。

- 定义节点和单元数据结构
- 定义 2D / 3D 网格容器
- 定义节点到全局 DOF 的映射规则

### 3. `stiffness`

负责单元级刚度计算。

- 按单元类型提供 `compute_*_element_stiffness` 函数
- 在单元内部处理材料参数、积分点、Jacobian、B 矩阵等逻辑
- 输出单元刚度矩阵，不直接处理全局装配

### 4. `assemble`

负责从单元刚度到全局刚度矩阵的装配。

- 接收统一的自由度映射和单元刚度计算回调
- 负责全局矩阵条目的汇总
- 不负责边界条件、不负责求解、不负责后处理

### 5. `boundary`

负责载荷和边界条件。

- 定义 2D / 3D 边界条件容器
- 构造全局载荷向量
- 施加 Dirichlet 约束
- 处理集中力、体力、边力、面力、重力

### 6. `solve`

负责线性方程组求解。

- 接收已经装配并施加边界条件后的矩阵与向量
- 输出位移解
- 不混入网格、后处理或业务层逻辑

### 7. `post`

负责结果导出和结果变换。

- 导出节点位移
- 导出单元应力和节点应力
- 导出 VTK
- 做极坐标等后处理变换

## 模块边界

下面这些边界应被视为硬约束。

### `mesh.py`

- 只定义核心数据结构和网格容器
- 不在这里写单元计算、边界条件装配或文件导出逻辑

### `dof_manager.py`

- 只负责自由度编号和映射
- 不混入材料、刚度、后处理逻辑

### `mesh_io.py`

- 只负责输入解析和对象构造
- 不应在这里做求解、装配或结果导出
- 允许做与输入格式相关的必要预处理，例如单元方向修正、材料参数映射

### `stiffness.py`

- 只负责单元级计算
- 不负责全局矩阵、载荷向量和结果导出
- 新单元优先以新增 `compute_*_element_stiffness` 的方式扩展

### `assemble.py`

- 只做全局装配
- 不在这里嵌入具体单元类型的业务分支

### `boundary.py`

- 只做边界条件和载荷相关逻辑
- 不在这里做求解或写结果文件

### `solve.py`

- 只做求解
- 不把边界条件、输入解析或导出逻辑塞进求解器

### `post.py`

- 只做结果导出和后处理
- 不在这里反向修改求解主流程

### `helper.py`

- 只放轻量级的几何选择辅助函数
- 不应演变成新的核心业务层

## 示例脚本的角色

`examples/*.py` 是本项目的实际使用入口，其职责是：

- 选择输入文件
- 调用 `mesh_io` 读取模型
- 组织装配、边界条件、求解和导出
- 作为用户侧参考流程存在

因此：

- 不要把复杂业务逻辑长期堆在示例脚本里
- 但也不要为了“看起来更工程化”而把示例脚本抽象成不必要的框架
- 如果某段逻辑已经被多个示例重复使用，再考虑下沉到 `src/fem`

## 允许的扩展方式

以下扩展通常是符合架构的：

- 在 `mesh_io.py` 中新增新的读取函数
- 在 `stiffness.py` 中新增新的单元刚度函数
- 在 `post.py` 中新增新的导出函数或结果变换函数
- 在 `helper.py` 中新增简单、明确的几何选择工具
- 在 `examples/` 中新增新的算例脚本

扩展时应遵循的原则：

- 优先复用已有数据结构和主流程
- 优先沿现有模块职责扩展，而不是新建平行层
- 新函数命名、参数风格和错误处理方式应尽量与现有代码一致

## 不允许或应谨慎对待的改动

以下改动默认视为架构风险较高，需要明确论证：

- 绕过现有主流程，直接在示例中写一套独立求解链
- 在多个模块中重复实现同一类单元逻辑
- 把装配逻辑塞入 `stiffness.py`
- 把求解逻辑塞入 `boundary.py` 或 `post.py`
- 把后处理逻辑塞入 `solve.py`
- 在没有整体设计的情况下，把项目半改成“安装包模式”
- 为单个问题引入新的大层次或新的抽象体系

## 新功能应放置的位置

常见新增内容的推荐落点如下：

- 新输入格式：`mesh_io.py`
- 新单元类型：`mesh.py`、`mesh_io.py`、`stiffness.py`、必要时 `post.py`
- 新载荷类型或约束类型：`boundary.py`
- 新结果导出格式：`post.py`
- 新几何选择工具：`helper.py`
- 新算例：`examples/`

如果一个功能跨越多个模块，也应保持沿现有主链路逐层接入，而不是另起一套结构。

## 不可随意破坏的接口和约定

以下内容默认应保持稳定：

- `src/fem` 下现有模块分工
- 网格对象的 `nodes`、`elements`、`num_dofs`、`global_dof`、`node_dofs`、`element_dofs`
- `compute_*_element_stiffness` 这类按单元提供的计算入口
- `assemble_global_stiffness_sparse`
- `BoundaryCondition2D` / `BoundaryCondition3D`
- `build_load_vector` / `build_load_vector_3d`
- `apply_dirichlet_bc` / `apply_dirichlet_bc_3d`
- `solve_linear_system_sparse`
- 现有 `examples` 的基本调用方式

如果确实需要改这些接口：

- 必须同步评估所有示例和下游调用的影响
- 必须在 review 中明确指出兼容性风险
- 不应做隐式破坏性修改

## 架构审查标准

当 agent 或协作者判断某次改动是否符合架构时，应至少检查：

- 是否仍沿主流程组织代码
- 是否保持模块职责单一
- 是否新增了不必要的平行实现
- 是否破坏了现有示例调用方式
- 是否错误假设项目已经包化安装

如果上述任一项答案明显为“是”，则该改动应被视为潜在架构问题。
